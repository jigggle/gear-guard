# Model Card: GearGuard Defect Risk Model

## Model Overview

**Model name:** GearGuard Defect Risk Model
**Project:** GearGuard: AI Quality Risk Monitoring for Football Equipment Operations
**Model type:** Binary classification
**Selected model:** Logistic Regression
**Primary task:** Predict whether a football equipment item is likely to be defective, fail inspection, or require removal from active rotation.

GearGuard uses equipment metadata, usage history, repair history, inspection scores, environmental exposure, and vendor-related features to estimate defect risk for football equipment operations.

The model is part of a broader quality monitoring workflow that includes:

* Defect-risk prediction
* Anomaly detection
* Drift monitoring
* Feature attribution
* Item-level quality triage summaries
* Streamlit dashboard reporting

---

## Intended Use

The model is intended to support equipment quality monitoring and inspection prioritization.

Appropriate uses include:

* Ranking equipment items by predicted defect risk
* Prioritizing items for manual inspection
* Identifying high-risk equipment before active use
* Supporting operational triage decisions
* Demonstrating manufacturing-style quality analytics in a football equipment context

The model is designed as a decision-support tool, not an automated replacement for human inspection.

---

## Not Intended For

The model should not be used as the sole basis for safety-critical decisions.

It should not be used to:

* Automatically approve equipment as safe
* Replace trained equipment staff or physical inspection
* Make real-world safety decisions without validated operational data
* Evaluate athlete performance or behavior
* Penalize athletes, staff, vendors, or teams
* Make procurement decisions without further analysis and real-world validation

Any high-risk prediction should be reviewed by a trained equipment manager or operations staff member.

---

## Data Limitations

The current dataset is synthetic and is designed for workflow demonstration.

Limitations include:

* The data does not come from real equipment-room records.
* Defect outcomes are simulated.
* Vendor, batch, and inspection patterns are artificially generated.
* The model has not been validated against real inspection outcomes.
* Feature relationships may be cleaner or more predictable than real operational data.
* The model should not be interpreted as a real-world equipment safety system.

A production version would require validated historical inspection logs, repair records, equipment metadata, usage tracking, staff notes, and verified defect outcomes.

---

## Modeling Approach

The project trains supervised models to predict `defect_reported`.

Models trained:

* Logistic Regression with class weighting
* Random Forest with class weighting

The selected model is Logistic Regression because it provided the best recall-focused performance on the test set.

The model uses a preprocessing pipeline with:

* Standard scaling for numeric features
* One-hot encoding for categorical features
* Class weighting to address class imbalance

---

## Evaluation Strategy

Because equipment defects are rare, raw accuracy is not the primary metric.

The project prioritizes recall because missing a defective item is more costly than sending an extra item for inspection.

Primary metric:

* Recall on defective items

Secondary metrics:

* Precision
* PR-AUC
* F1-score
* ROC-AUC
* Confusion matrix

Current test metrics:

| Metric             | Value |
| ------------------ | ----: |
| Recall             | 0.848 |
| Precision          | 0.179 |
| PR-AUC             | 0.509 |
| Selected threshold |  0.50 |

---

## Operational Trade-Off

The model is intentionally recall-focused.

This means:

* It catches most defective items.
* It also produces false positives.
* Some non-defective items may be sent for inspection unnecessarily.

This trade-off is acceptable for the project because the operational cost of a missed defect is assumed to be higher than the cost of an additional inspection.

In a real deployment, the threshold should be reviewed with equipment staff and adjusted based on inspection capacity, risk tolerance, and historical defect severity.

## Model Outputs

For each equipment item, the model produces:

* `predicted_defect_risk`
* `defect_risk_flag`
* `risk_tier`
* `recommended_action`

Risk tiers are used to make the output easier to interpret in the dashboard.

Example risk tiers:

* `critical`
* `high`
* `medium`
* `low`

Example recommended action:

```text
Remove from active rotation and inspect immediately.
```

## Explainability

The project includes item-level feature attribution through `src/explain.py`.

For the selected Logistic Regression model, explanations are computed from standardized feature values multiplied by model coefficients. This provides directional feature attribution for why a specific item was assigned a high or low risk score.

Example explanation:

```text
EQ-00206 (shoulder_pads) was flagged as critical risk with predicted defect probability of 100.0%. The strongest risk drivers were: repair_count, inspection_score, material_condition_score, game_exposure_count. Recommended action: review inspection history, verify physical condition, and compare against similar items from the same equipment type or vendor.
```

Outputs:

* `reports/feature_importance.csv`
* `reports/item_explanations.json`

## Anomaly Detection

The project also includes an Isolation Forest anomaly detection model.

The anomaly model is separate from the supervised defect-risk model. It identifies unusual equipment quality signatures based on feature patterns.

Current anomaly configuration:

* Model: Isolation Forest
* Contamination rate: 5%
* Output: `anomaly_score` and `anomaly_flag`

Current result:

| Metric       | Value |
| ------------ | ----: |
| Anomaly rate | 5.00% |

Anomaly detection is useful for identifying unusual records even when a supervised defect label is not available.

## Ethical and Operational Considerations

The model should support better inspection workflows, not replace human judgment.

Important considerations:

* A high-risk score does not prove an item is defective.
* A low-risk score does not guarantee an item is safe.
* Synthetic data limits real-world validity.
* Human inspection remains required.
* The model should be reviewed before any real-world use.
* Thresholds should be adjusted based on operational capacity and risk tolerance.
* Model outputs should be communicated clearly to avoid overconfidence.

---

## Monitoring Recommendations

If this system were deployed with real data, the following should be monitored:

* Defect rate over time
* Recall on confirmed defective items
* False positive inspection burden
* Vendor-level defect trends
* Equipment-type-level defect trends
* Drift in key features
* Changes in inspection score distributions
* Changes in predicted risk distributions
* Model performance by equipment type
* Model performance by season or vendor batch

The model should be retrained if performance drops or if quality signals drift significantly.
